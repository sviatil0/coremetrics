Date: 2026-06-18
Auditor: review pass against origin/dev tree
Scope:

- `coremetrics.cpp`
- `src/*.cpp` and `include/*.hpp`
- `tests/*.cpp` and `bench/*.cpp` (sampled)
- `scripts/*.sh`
- `.github/workflows/*.yml`

## High findings

### H1. base.xml `std::stof` parser throws uncaught on malformed input

- File: `src/GUIFile.cpp:60-61`, `:77-79`, `:92`, `:212-214`, `:240`
- A hostile or merely corrupted `base.xml` (the file is shipped in the tarball and lives next to the binary, but is plaintext and editable) reaches `std::stof` with no `try/catch`. Any token that is not a parseable float (empty, non-numeric, overflowing, e.g. `<x>1e+999</x>`) raises `std::invalid_argument` or `std::out_of_range`. There is no top-level `catch` around `GUIFile::readFile` so the unhandled exception terminates the process at start-up.
- Remediation: wrap each numeric extraction in a helper that catches `std::exception` and falls back to a documented default (0.0f); or call `std::strtof` with a `char**` end pointer and validate `errno`.
- Sample fix:
  ```cpp
  static float safeStof(const std::string &s, float fallback = 0.0f)
  {
      try { return std::stof(s); }
      catch (...) { return fallback; }
  }
  ```

### H2. base.xml `recurseLayout` is unbounded recursion (stack DoS)

- File: `src/GUIFile.cpp:122-148`
- `recurseLayout` recurses for every nested `<layout>...</layout>` it finds and there is no depth cap. A pathological `base.xml` with thousands of nested layouts blows the C++ call stack and crashes the program. Same input class as H1 (attacker who can replace `base.xml`).
- Remediation: pass `int depth` and reject when `depth > kMaxLayoutDepth` (suggest 64 or 128); record a one-line warning and return the parent untouched.

### H3. `pull_request_target` workflow checks out untrusted PR head

- File: `.github/workflows/pr-size.yml:1-37`
- The trigger is `pull_request_target`, and the workflow checks out `${{ github.event.pull_request.head.sha }}` (line 14). `pull_request_target` runs with the upstream `GITHUB_TOKEN` and has `pull-requests: write`. The job only runs `git diff --stat` and posts a comment, so the immediate attack surface is small, but the established anti-pattern (privileged token plus untrusted code on disk) is exactly how the 2021 GitHub-Actions worm class of bugs starts. A future contributor adding `make` or any build step here would instantly escalate to RCE on the upstream runner.
- Remediation: switch to `pull_request` (token is automatically lowered to fork-safe perms), and read `BASE_SHA` / `HEAD_SHA` from the `pull_request` event payload instead.

### H4. CI workflows pin third-party actions to floating `@main`

- File: `.github/workflows/c-cpp.yml:49, 58, 241, 289`, `.github/workflows/cpp-lint.yml:75`, `.github/workflows/release.yml:50`
- Every reference to `libsdl-org/setup-sdl@main` runs whatever code is currently on that repository's default branch. A compromise of `libsdl-org/setup-sdl` (or a maintainer mistake) lands instantly in this repo's `windows`, `macOS`, and `Linux` release builds, including the **release.yml** job that ships the signed tarball. The same risk applies more weakly to `actions/labeler@v5`, `actions/stale@v10`, `softprops/action-gh-release@v2`, `raven-actions/actionlint@v2`, and `orhun/git-cliff-action@v3`, which are pinned to a major-version float and can be moved by the action's owner.
- Remediation: pin every third-party `uses:` to a 40-char commit SHA (`uses: libsdl-org/setup-sdl@<sha>`). Dependabot's `package-ecosystem: github-actions` updater (`.github/dependabot.yml`) will keep them current.

## Medium findings

### M1. `Notification::post` shells out via `std::system`

- File: `src/Notification.cpp:91, 111`
- The macOS and Linux branches build a shell command and invoke `std::system(command.c_str())`. Title and message are escaped via `shellEscapeForSingleQuote` and (on macOS) `appleScriptStringEscape`, and the audit confirms the escapers are correct for the documented contexts. The residual exposure is that the title/message come from CoreMetrics' own threshold-breach formatter today, but the function is a `public` API that any future call site could feed user-supplied or process-name-derived text into. Preferable to drop the shell entirely.
- Remediation: replace with `posix_spawn` / `execvp` (Linux: `notify-send`, macOS: `osascript -e <script>`) so arguments never go through a shell parse. Eliminates the entire escaping correctness burden.

### M2. Font glyph surface cache is unbounded

- File: `src/font.cpp:105-107` (the three `g_textSurfaceCache*` maps)
- One `SDL_Surface*` is allocated per unique `(text, RGB)` tuple per font size, with no eviction. Process names are rendered through this cache (the table refreshes every poll). A long-lived session that watches a system spawning many short-lived processes with unique names accumulates one surface per name forever. Memory leak / growth indistinguishable from a slow-leak DoS on long sessions.
- Remediation: bound each map to e.g. 4096 entries with simple LRU eviction (a `std::list` plus an `unordered_map<key, list-iterator>` is enough). Destroy the evicted surface before erasing.

### M3. `Image::draw` re-loads and re-decodes the file every frame

- File: `src/image.cpp:14-52`
- `IMG_Load` runs on every `draw()` call, which is once per frame (~60Hz). A path read from `base.xml` (loaded via `GUIElementFactory::Image` per `GUIFile.cpp:261`) silently re-opens the file at refresh rate. Beyond performance, this becomes a denial-of-service primitive against a network-backed `--filesystem=host-os:ro` flatpak mount, and the same code path can be pointed at `/dev/zero` or a fifo to lock the render thread.
- Remediation: decode once in the constructor, store the surface in a `unique_ptr`-managed handle, and blit on draw.

### M4. PID race window between selection and signal send

- Files: `src/KeyboardEvents.cpp:202`, `src/SignalUtils.cpp:65-87`
- The kill flow stores `g_selectedPid` when the user opens the signal menu, then sends the signal after the user confirms. Between those two events a process can exit and the kernel can recycle the pid number. The poller refresh (`MetricsPoller.cpp:521-526`) drops the selection if the pid disappears from `g_visiblePids`, but that refresh only fires on the poll cadence (default 500 ms; up to 10 s), and the keyboard handler does not re-check before calling `SignalUtils::send`. Practical attack: a malicious local actor cycles a pid namespace to land their preferred pid on a target value while the user is reading the confirmation dialog. Cross-platform: same window on Windows where `OpenProcess` would succeed on whatever new owner holds that pid.
- Remediation: at confirm time, re-validate that `g_selectedPid` is still in `g_visiblePids` AND that the row at `g_selectedRowIndex` still maps to the same pid AND (Linux) that `/proc/<pid>/start_time` matches a snapshot captured at menu-open time. Reject otherwise with a "stale selection" status flash.

### M5. `c-cpp.yml` has no workflow-level `permissions:` block

- File: `.github/workflows/c-cpp.yml:1-30`
- Without an explicit `permissions:` block the workflow inherits the repository default token scope, which on public repos is currently `contents: read` but can silently widen if the org default is later relaxed. The matrix runs untrusted-by-design build code (full make, brew install, etc.), so an explicit ceiling protects against future drift.
- Remediation: add a top-level `permissions: {contents: read}` block. The matrix jobs and the `sanitizers` / `coverage` reusables do not need write.

### M6. `brew-tap-publish.yml` uses `git push --force-with-lease` against a downstream main-targeted PR branch

- File: `.github/workflows/brew-tap-publish.yml:175`
- Force-pushing is correct for the per-tag PR branch, but the BREW_TAP_TOKEN is a long-lived PAT (lines 9-12 of the file's docstring) with `contents: write` + `pull-requests: write` on `sviatil0/homebrew-coremetrics`. Long-lived PATs are a known supply-chain anti-pattern. The workflow runs on `workflow_run`, which is itself a trigger that runs on the *upstream* default branch's workflow definition, so a compromise of dev â†’ the workflow can pivot to the tap repository immediately.
- Remediation: replace BREW_TAP_TOKEN with a GitHub App installation token (`actions/create-github-app-token@<sha>`), or at minimum use a fine-grained PAT scoped only to `sviatil0/homebrew-coremetrics` with `Contents: Read and write`, `Pull requests: Read and write`, no other scopes, with a 90-day expiry.

### M7. `IMG_Load` icon path is silently re-read on every launch with no integrity check

- File: `coremetrics.cpp:562-567`
- `AssetPath::resolve("assets/logo.png")` returns the resolved path, then `IMG_Load` runs against it. If the assets directory is writable by another local user (it is, on Linux installs that drop into `/usr/share/coremetrics/assets` then bind-mount through Snap or Flatpak with weak permissions; or on Windows where `C:\Program Files (x86)\coremetrics\assets` can be ACL-flipped), an attacker can replace `logo.png` with a maliciously crafted PNG and any libsdl3_image CVE becomes locally exploitable. Same pattern for `assets/font.ttf` loaded via `font.cpp:84`.
- Remediation: ship asset SHA-256 sums in a sidecar file and verify them before load. Lower-cost alternative: install the assets read-only (mode 0644, owned by root) and document the permission expectation in `packaging/README.md`.

### M8. `lint.yml` actionlint job runs with `continue-on-error: true` and `fail-on-error: false`

- File: `.github/workflows/lint.yml:23, 36`
- Findings from actionlint (e.g. a maintainer accidentally introducing `pull_request_target` plus `checkout @ head.sha` in a new workflow) are advisory-only. The workflow surfaces shellcheck issues in the job log but never gates a PR. This is a process-hardening concern rather than a code finding; flagged here because actionlint is the closest CI proxy for catching H3-class regressions.
- Remediation: once the existing actionlint backlog is cleared, flip both `continue-on-error` and `fail-on-error` to `false`.

## Low findings

### L1. `ArgParse` uses `std::atof` / `std::atoi` instead of `std::strtoul`-style parsers

- File: `src/ArgParse.cpp:84, 137`
- `std::atof("garbage")` returns `0.0` with no error indication, and `std::atoi` returns `0` on overflow. The follow-on clamps (`< 0.0 -> 0.0`, `< 1 -> 20`, `> 999 -> 999`) absorb the worst cases, so this is Low. `--poll-ms` already uses the safer `clampPollIntervalMs` helper (`src/ProcessUtils.cpp:168`) which is the right pattern.
- Remediation: rewrite `--duration` and `--top` parsing to mirror `clampPollIntervalMs` (strtoull + explicit overflow/negative rejection).

### L2. `screenshotPath` echoed verbatim to stdout discloses the full path

- File: `src/HeadlessRunner.cpp:625`
- `"Saved screenshot to " << screenshotPath` prints whatever the user passed. On shared CI logs or terminal recordings this can disclose the user's `$HOME` or working directory. Not exploitable on its own; tagged because the project ships with shareable smoke-screenshot logs.
- Remediation: print only the file basename, or gate the path echo behind a `--verbose` flag.

### L3. `IMG_Load`-based asset path swallows error, prints to stderr

- File: `src/image.cpp:19`, `src/font.cpp:66, 87`
- These log to `std::cerr` and continue. A red-team operator who can write to the working directory could bait the user into running CoreMetrics from a folder where assets fail to load, leaving the UI broken silently. Defensive only; the CoreMetrics process never elevates here.
- Remediation: track a single `g_assetLoadFailure` flag and surface a one-line status flash on the live UI so the user notices.

### L4. `stale.yml` cron job has broad write permissions

- File: `.github/workflows/stale.yml:8-11`
- `issues: write` + `pull-requests: write` is exactly what `actions/stale@v10` needs, so the scope is correct. Flagged Low only because the action itself is pinned to `@v10` (a movable tag â€” see H4) and the daily cron means any compromise of `actions/stale` is on a tight blast-radius timer.
- Remediation: SHA-pin per H4.

## Informational notes

### I1. Signal handler is async-signal-safe

`handleShutdownSignal` only does `g_shutdownRequested.store(true)` on a `std::atomic<bool>`, with no allocation, locking, or I/O. Confirmed at `coremetrics.cpp:131-134` and `src/HeadlessRunner.cpp:386-387`. Correct.

### I2. ThreadPool lifetime + lambda capture safety

`Screen::drawBox` and `Screen::drawTriangle` (`src/screen.cpp:284, 328`) capture `this` plus copies of all relevant ints/colors by value, then call `pool.submit(...)` and `f.get()` synchronously in the same frame. No reference capture, no escaping future. Correct.

### I3. Sparkline storage is bounded

`RingBuffer<T>` (`include/RingBuffer.hpp`) is a fixed-capacity circular buffer with no per-push allocation. `Sparkline::push` writes through it. No unbounded growth.

### I4. `KillLog` and `TraceLog` writers are RAII-scoped

Both use `std::ofstream` member or stack instances and let the destructor close. `TraceLog` additionally guards `g_stream` under `g_mutex` and double-checks the active flag after acquiring the lock (`src/TraceLog.cpp:131-139`). Correct.

### I5. Settings parser is exception-free and bounds-checked

`src/Settings.cpp:120-142` implements `parseUnsigned` by hand with explicit overflow check, then bounds-clamps each persisted value (`POLL_MS_MIN/MAX`, `SORT_COLUMN_MIN/MAX`, `PID_MAX`). Bookmark and KillLog mirror the pattern with manual digit checks. Good defensive shape; can serve as the reference for the H1 / L1 rewrites.

## Out of scope but worth a follow-up

- `actions/setup-sdl@main` itself: file an upstream issue with `libsdl-org/setup-sdl` asking them to publish SHA-tagged releases. Right now every consumer of that action has a soft supply-chain dependency on the project's `main` branch.
- The Snap manifest declares `plugs: [..., system-observe, process-control]` (`snap/snapcraft.yaml:36-37`). `process-control` is the broad signal-sending plug; this is correct for CoreMetrics' kill flow but is an auto-connection negotiation with the Snap Store. Document explicitly in `packaging/README.md` so a reviewer at the Snap Store understands the scope request.
- Flatpak manifest declares `--filesystem=host-os:ro` (`packaging/flatpak/io.github.sviatil0.CoreMetrics.yml:14`). This is the minimum needed for `/proc` and `/sys` access on the Processes tab. Document the rationale alongside the manifest so Flathub reviewers do not bounce it.
- The chocolatey install script (`packaging/chocolatey/tools/chocolateyinstall.ps1`) ships with `REPLACE_WITH_WINDOWS_ZIP_SHA256` as a checksum literal. Until the Windows release artifact is wired into the release workflow, this package will fail at install time, which is the right behavior. Track in an issue so it does not silently regress to a permissive checksum.
- The release pipeline does not publish a side-by-side `coremetrics-vX.Y.Z-*.tar.gz.sha256` file alongside each artifact. The brew tap publisher computes sha256 internally, but a manual `curl | tar xz` user has no integrity reference. Add a `sha256sum *.tar.gz *.deb > SHA256SUMS` step to `release.yml`.
- No SLSA / Sigstore / cosign attestation is published with the release artifacts. Adding `actions/attest-build-provenance@<sha>` to `release.yml` is a small change that unlocks supply-chain provenance for downstream packagers.

## Recommendations

In order of effort vs. reward:

1. **Pin every third-party action to a SHA** (H4). One-line per `uses:`; closes the largest supply-chain hole. Dependabot already configured.
2. **Wrap `std::stof` / `std::stoi` in `try/catch` in `GUIFile.cpp`** (H1). Half a day; eliminates a crash class.
3. **Cap `recurseLayout` depth at 128** (H2). Two-line change; eliminates a DoS class.
4. **Switch `pr-size.yml` from `pull_request_target` to `pull_request`** (H3). One-line trigger change; eliminates the privileged-token-on-untrusted-code anti-pattern.
5. **Add LRU eviction to the font glyph cache** (M2). ~50 lines; closes the long-session memory growth.
6. **Cache decoded image surface in `Image` constructor** (M3). ~30 lines; performance win plus closes the per-frame I/O channel.
7. **Re-validate pid at signal-confirm time** (M4). One read + one bool check; closes the pid recycle race.
8. **Add a workflow-level `permissions: contents: read` to `c-cpp.yml`** (M5). Two-line change.
9. **Replace `Notification::post` `std::system` with `posix_spawn`** (M1). Half a day; removes the entire shell escaping surface.
10. **Migrate `BREW_TAP_TOKEN` to a GitHub App installation token** (M6). Half a day plus secret rotation.
11. **Verify asset checksums on load** (M7). One day; protects against the ACL-flip class.
12. **Add `SHA256SUMS` publication + Sigstore provenance to `release.yml`** (informational follow-up). Half a day; downstream consumers get integrity guarantees.

This audit is a snapshot at `origin/dev` for the 2026-06-18 review window. Subsequent PRs will address findings individually; no code changes were made as part of this report.
